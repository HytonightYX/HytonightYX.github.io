# 为什么学习前端呢？

我觉得自己大学做过的东西还是比较多的，大一学了半年的AI、大二上学期自学了Java后端开发，而到了大二下学期正式接触前端开发，我发现前端开发才是我的菜。

为什么？

首先是因为喜欢，当时自己做后端，但是也要写前端呀，用的那个套模板的方式开发，说实话做出来的东西不好看，而且开发很繁琐，不好维护，当然这已经是一种比较古老的开发方式了。那我是一个，emm，有点小强迫症的，所以这驱使我去了解前端最新的开发方式，自然而然了解到了React，Vue这样的前端工程化开发框架，React 一上手我就很喜欢，进而去了解了前端的发展史，然后就正式入坑，做起了React + Java 的全栈开发。那后来随着 js 用的越来越熟练，我也越来越喜欢了，所以干脆弃坑 Java，用上了Node.js。算是正式入坑大前端领域了吧。

但是我之前学的AI，学的Java就没用了吗？当然不是，我学的后端，让我有了架构的思维，有了数据库设计的经验，有了开发规范的经验，所以这给我后来商业项目的全栈开发打下了很好的基础；学习过AI，tensorflow，所以我能够独立做出整个打卡模块，因为打卡模块里就有一个人脸识别的过程。

第二呢就是我在学习的过程中慢慢意识到，前端有拓展未来全栈的潜力，因为Node的出现，推动了前端的工程化以及后端开发能力。

所以我觉得这几年Web开发领域，因为用户对于“体验”二字的的要求越来越高，所以推动了前端的发展，是一个优化体验的过程，也是一个优化管理的过程。

# 怎么学习前端呢

# 广度：

关注社区，紧跟潮流：比如国内的前端团队博客、掘金，国外的Medium，Youtube，telegram讨论组；

我看到国外一些比较好的文章，我会进行一个翻译并且通过掘金翻译计划这个开源项目来发布。目前是有2篇的成果。

项目产生问题，谷歌解决方案；

订阅周刊，JS Weekly，React Status，他们会把每周精选的一些好文章，库的更新，大牛的讨论等等汇总发到邮箱；

# 学习新技术

首先是看官方文档的前几篇，搞懂这是什么技术栈，能解决什么问题。如果有官方小例子也可以马上实践一下。

找入门视频看，B站，YouTube，做一个小项目

学习最佳实践：谷歌 技术名 + Best Practice

GitHub 搜索真实项目，看看如何和其他技术栈配合实现一个项目，这样就差不多了

然后看有没有实际的使用场景，比如刚好自己的项目里可以用上

总的来说是项目驱动吧，实践为主

# 深度：

系统性地看书，看博客。

看源码，配合一些源码解析的文章，慢慢深入理解

学习设计思路，尝试自己实现一个最小化框架（防御性代码）

从一个高的层面入手，慢慢细分，慢慢钻研。

坚持做输出，发布到社区，一方面是我想去将知识进行分享，帮助别人；另一方面也是为了更好地理解学过的内容；

# 讲一讲对你影响很大的人

我的导师：他非常善于剖析问题和简化问题，用最简洁的方法解决问题。之前我说的状态机优化就是我请教我的导师后解决的。这和做架构设计其实也是一样的，大道至简。我在刚开始做系统设计的时候就会关注很多的细节，结果造成了过度设计，但是在导师的指点下我慢慢也学会简化问题，进而更好地设计系统。

# 对个人未来发展的看法

我在大二给自己定下了人生的三个梦。大厂梦，架构梦，开源梦。这都是每3-5年的阶段性目标。

而现在就是实现我大厂梦的时候，这两年我一直为了这个目标而努力。

而架构梦，我想在实现了大厂梦之后的3-5年里能成为前端架构师。

开源梦，就是我希望在有生之年能够成为像yyx那样的大神，做一个开源库，推动前端领域的发展。

架构梦

原因呢是我在写一个小项目的时候偶然发现了“设计模式”这四个字，在谷歌上探索了一番，把好几个设计模式都实践了一下，突然感觉发现了新大陆。这都是前人总结的最佳实践呀

就是我认为软件工程是一种美学，因为当我去回顾计算机发展的历史，软件工程发展的历史，我突然发现一个事情，那就是，建立在那些底层的数字公式、0和1的二进制运算之上的，是人性。

前端架构设计，往前看是为了提供更好的用户体验，往后看，是为了方便地进行开发，所以有了高内聚、低耦合，另外人月神话中说的，软件开发没有银弹。

所以前端是我决定深入的一个方向，扎根与前端领域，深入前端，精通前端，最后呢，跳出前端，我想在3到5年之后能承担整条业务线的系统设计，成为大前端架构师。我知道就是架构师也是有级别的，比如最高的一级架构师，以及若干子系统的二级架构师。
